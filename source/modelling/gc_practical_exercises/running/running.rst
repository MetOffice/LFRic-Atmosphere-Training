***************************
Running a Rose Workflow
***************************

This page describes a simple end-to-end process for running a Rose workflow. It covers checking out the training suite, configuring it for your platform, launching the workflow, monitoring its progress, and cleaning a redundant workflow.

Checking out the training suite
===============================

Create a working area and check out the suite:

.. code-block:: bash

	# If it doesn't exist already, create a directory for rose suites
    mkdir -p ~/roses
    # cd into that directory
    cd ~/roses
    # copy the training suite
	rosie cp <SUITE-ID>

Configuring the run
===================

Open the suite configuration (``rose edit``) and adjust platform-specific settings:

.. code-block:: bash

	rose edit

Typical items to check:

* Platform/site selection.
* The rose_suite.info file which specifies the suite metadata, including the description, owner, project, and title.
* Build and run queue options.
* Wallclock limits.

Save and close the editor when complete. You do not need to use ``rose edit`` to make changes to the suite configuration, you can also edit the files directly. The ``rose edit`` command is just a convenient way to open the relevant files in an editor.

Installing and launching the workflow
=====================================

Install and run the suite:

.. code-block:: bash

	cylc vip -O <SITE_FILE>

This command uses Cylc to Verify, Install, and Play (VIP) the workflow. Replace ``<SITE_FILE>`` with the site configuration file for your platform.

Monitoring progress
===================

Use one of the below Cylc commands to monitor task states:

.. code-block:: bash

    cylc tui   # Terminal user interface

.. code-block:: bash

    cylc gui  # Graphical user interface

They both show nearly identical information and which you use is down to preference. The gui is easier to interact with but does require an X11 connection however the tui is more lightweight and works well in a terminal.

Verifying the workflow has run successfully
===========================================

After successful completion of the workflow, you should see all tasks marked as ``succeeded`` in the monitoring interface. You can also inspect the output files generated by the model in the suite run directory to confirm that expected diagnostics or fields are present.

As a check:
* Confirm all required tasks are ``succeeded``.
* Inspect model output in the suite run directory.
* Validate expected files, fields, or diagnostics from the exercise.

Stopping or cleaning a workflow
===============================

To stop a running workflow:

.. code-block:: bash

	cylc stop <SUITE-ID>

You do not need to wait for the workflow to stop before removing it.

To remove an installed workflow:

.. code-block:: bash

	cylc clean <SUITE-ID>

This will remove the workflow from the system but the suite directory will still exist in ``~/roses``. You can remove this manually if you wish.
